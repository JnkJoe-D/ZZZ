# 动作指令中枢重构落地方案

## 1. 文档目标

本文档用于为当前项目落地一套面向《绝区零》复刻目标的动作指令中枢重构方案。

目标不是做“理论上最优”的全新框架，而是：

- 保留当前已经可用、手感较好的 `ComboWindow` 时间轴配置能力
- 让所有非受击动作的播放最终都收敛到统一中枢调用
- 解决当前地面移动、非连段技能、Dash 派生仍然绕过中枢的问题
- 支持不同 Dash 状态接不同的冲刺普攻表现
- 让系统从“技能连段控制器”升级为“全动作指令路由器”
- 为后续 `ActionGraph` / `ComboGraph` 的角色级可视化管理预留稳定数据结构

---

## 2. 结论先行

### 2.1 核心结论

当前项目真正缺的不是更多 `ActionConfig`，而是一个独立于播放资源的：

`指令上下文层 (Command Context Layer)`

应当将系统拆分为三层：

1. `Command`
   - 描述玩家或系统发起的动作意图
   - 轻量、统一、可缓冲
2. `Context`
   - 描述角色当前处于什么逻辑上下文
   - 决定哪些命令在当前时刻有资格被接收
3. `Route`
   - 描述命令在某个上下文或某个动作窗口内命中后，应该跳到哪里

简化表述如下：

- `ActionConfigAsset` 负责“播什么”
- `CommandContextConfig` 负责“当前允许什么”
- `CommandController` 负责“唯一裁决并调用播放”

### 2.2 对当前 `ComboWindow` 的态度

`ComboWindow` 不但不该废弃，反而应该升格为：

`动作局部指令窗口系统`

它继续负责动作内部、时间敏感的派生，例如：

- 普攻连段
- Dash 冲刺普攻
- 特殊技变体
- 弹反成功后的追击
- 仅在某动画区间有效的取消

### 2.3 对 Dash 的最终建模

按《绝区零》复刻目标，建议最终语义如下：

- `Evade` 是玩家命令
- `FrontEvade / BackEvade` 是解析结果，不是核心命令类型
- `Dash` 是前闪后的自动续态，不是独立玩家输入指令
- `DashAttack` 是 Dash 动作自身的局部派生，不是全局常驻入口

这意味着：

- `Dash` 的普通技能入口不应依赖 `GetActionState(Dash)`
- `DashAttack` 应优先在当前 Dash 动作的 `ComboWindow + LocalRoutes` 中配置
- Ground 默认入口只负责“进入 Dash 上下文”，不负责“决定冲刺普攻”

## 3. 当前系统的核心问题

### 3.1 输入链路仍然是双中枢

当前输入先进入当前状态的 `IInputCommandHandler`，再决定是否转发给 `ComboController`。

这导致：

- 技能态、闪避态使用 `ComboInputCommandHandler`
- 地面态、冲刺态、停止态仍使用 `DefaultInputCommandHandler` / `DashInputCommandHandler`
- 非连段时的技能与闪避入口仍然是手写 `NextActionToCast + ChangeState`

结果是：

- 技能连段由中枢处理
- 非连段时的技能入口由状态处理
- locomotion 与 skill 派生体系割裂

### 3.2 控制器只认 Skill，不认通用 Action

虽然 `ActionConfigAsset` 已经在基类层拥有 `OutTransitions`，但运行时控制器内部仍把当前动作强转为 `SkillConfigAsset`。

这导致：

- `LocomotionConfigAsset` 无法真正参与局部派生判断
- Dash 时间轴上的 `ComboWindow` 即使触发，也不会被当作有效局部路由消费
- `DashStart` / `DashLoop` / `DashTurnBack` 只能继续由子状态手动驱动

### 3.3 当前动作与当前逻辑上下文被混为一谈

这是当前设计最本质的问题。

例如：

- `DashStart`
- `DashLoop`
- `DashTurnBack`
- `DashStop`

它们是不同播放动作，但本质上都属于更高层的：

`GroundDash Context`

如果继续让每个播放动作都承担完整逻辑上下文职责，会导致：

- 配置分散
- 派生入口重复
- 高扇出状态资产爆炸
- 逻辑权限和动画表现耦合过深

### 3.4 `BufferedInputType` 混合了语义、方向和阶段

目前枚举中同时混有：

- 指令语义：`BasicAttack` / `SpecialAttack` / `Ultimate`
- 方向变体：`EvadeFront` / `EvadeBack`
- 长按阶段：`BasicAttackHold` / `BasicAttackHoldCancel`

这会导致两个问题：

1. 枚举膨胀
2. 语义边界不清晰

一旦未来加入：

- 左右闪
- 空中闪
- 锁定闪
- 特殊技长按不同阶段
- 完美时机派生

当前枚举必然快速膨胀。

## 4. 目标架构

## 4.1 总体流转

```text
InputProvider / AI / System Event
        ↓
CommandAdapter
        ↓
CharacterCommand
        ↓
CommandBuffer
        ↓
CommandController
        ├─ 读取 CurrentAction 的 LocalRoutes
        ├─ 读取 CurrentContext 的 ContextRoutes
        ├─ 解析 Resolver / Conditions
        └─ 唯一调用 ActionPlayer.PlayAction
        ↓
StateMachine / Context 切换
```

### 4.2 职责划分

#### InputProvider

职责：

- 读取原始输入
- 提供方向向量、按键事件、长按状态
- 不直接参与动作选择

#### CommandAdapter

职责：

- 把输入事件翻译成统一命令
- 统一方向、阶段、时间戳
- 处理输入层去重和优先级折叠

例如：

- `BasicAttack.started -> Command(BasicAttack, Started)`
- `SpecialSkill.started -> Command(SpecialAttack, Started)`
- `EvadeFront / EvadeBack -> Command(Evade, Started, DirectionSnapshot)`

#### CommandBuffer

职责：

- 只缓存离散命令
- 负责生存期与消费标记
- 不处理业务裁决

注意：

- 连续移动向量不应每帧入缓冲
- 缓冲的应当是边沿事件，例如 `MoveStarted` / `MoveStopped`

#### CommandController

职责：

- 唯一动作裁决中枢
- 唯一 `ActionPlayer.PlayAction` 调用者
- 唯一状态切换协调者
- 统一消费 `LocalRoutes + ContextRoutes`

#### State / SubState

职责：

- 维护位移、物理、上下文切换
- 不再直接决定技能入口
- 不再直接决定 `ActionPlayer.PlayAction`

#### ActionPlayer

职责：

- 纯播放器
- 不做业务判断
- 不决定上下一个动作

## 5. 数据结构设计

### 5.1 统一命令结构

```csharp
public enum CommandType
{
    MoveStarted,
    MoveStopped,
    BasicAttack,
    BasicAttackHold,
    SpecialAttack,
    SpecialAttackHold,
    Ultimate,
    Evade,
}

public enum CommandPhase
{
    Started,
    Performed,
    Canceled,
}

public struct CharacterCommand
{
    public CommandType Type;
    public CommandPhase Phase;
    public float Timestamp;
    public Vector2 DirectionSnapshot;
    public bool IsConsumed;
}
```

### 5.2 逻辑上下文枚举

```csharp
public enum CommandContextType
{
    GroundIdle,
    GroundJog,
    GroundDash,
    GroundStop,
    Skill,
    Evade,
    Backswing,
    HitStun,
}
```

### 5.3 路由数据结构

建议将现有 `ComboTransition` 升级为通用 `ActionRoute`：

```csharp
public enum RouteScope
{
    LocalAction,
    Context,
}

[Serializable]
public class ActionRoute
{
    public CommandType RequiredCommand;
    public CommandPhase RequiredPhase;
    public string RequiredWindowTag;
    public ComboTriggerMode TriggerMode;
    public Config.ActionConfigAsset NextAction;

    [SerializeReference]
    public List<ITransitionCondition> ExtraConditions = new();
}
```

```csharp
    public Config.ActionConfigAsset DefaultEnterAction;
}
```

## 6. 路由分层规则

### 6.1 LocalRoutes

挂在 `ActionConfigAsset` 上。

适合配置：

- 普攻连段
- Dash 冲刺普攻
- 某技能特有的取消
- 某动作内的特殊分支

判断前提：

- 当前播放动作就是该 Action
- 符合时间轴窗口要求
- 满足指令、阶段、条件

### 6.2 ContextRoutes

挂在 `CommandContextConfig` 上。

适合配置：

- Idle 时按普攻起手
- Jog 时按特殊技
- Ground 时按终结技
- 非受击时按闪避
- Evade 结束后前闪续 Dash

判断前提：

- 当前逻辑上下文匹配
- 不要求必须存在某个局部动作窗口

### 6.3 最终优先级

建议中枢始终按以下优先级裁决：

1. `HitStun / Hard Lock`
2. 当前动作 `LocalRoutes`
3. 当前上下文 `ContextRoutes`
4. 默认自动续播/自动回流

这样可以保证：

- DashAttack 优先命中 Dash 的局部窗口
- Idle / Jog 又不会因为没有窗口而无法起手
- 普攻连段、冲刺普攻、闪避追击能稳定共存

## 7. Dash 与冲刺普攻的标准落地方式

### 7.1 推荐建模

推荐保持下面这套语义：

- 玩家按闪避，产生 `Evade` 命令
- `EvadeResolver` 根据方向解析成前闪或后闪动作
- 如果解析为前闪，则在动作结束或回地时写入 `ForceDashNextFrame`
- `GroundContext` 检测到该标记后切入 `GroundDash`
- Dash 的冲刺普攻由 Dash 动作自己的 `LocalRoutes` 决定

### 7.2 为什么 DashAttack 应优先放在 LocalRoutes

因为它满足三个条件：

1. 依赖当前正在播放的 Dash 动作
2. 依赖时间窗口
3. 不同 Dash 状态可能接不同 DashAttack

这正是 `LocalRoutes` 最擅长的事情。

### 7.3 标准配置范式

以 `DashLoopConfig` 为例：

Timeline：

- 配一个 `ComboWindowClip`
- `ComboTag = DashAttack`

Routes：

- `RequiredCommand = BasicAttack`
- `RequiredPhase = Started`
- `RequiredWindowTag = DashAttack`
- `TriggerMode = InstantOnly`
- `NextAction = 对应 DashAttack`

如果角色存在多个不同 Dash：

- `DashLoopA -> DashAttackA`
- `DashLoopB -> DashAttackB`

则分别配置各自的 `LocalRoutes` 即可。

### 7.4 关于 “instantly” 的落点

建议不要将 “InstantOnly” 设计成窗口类型。

更清晰的做法是：

- `Window` 只决定时机范围
- `TriggerMode` 决定是否允许预输入

因此 Dash 冲刺普攻应当配置为：

- `Window = Execute`
- `Route.TriggerMode = InstantOnly`

这样概念更统一，编辑器也不需要引入额外窗口类型。

## 8. 对现有 ComboWindow 系统的保留与升级建议

### 8.1 保留

保留当前：

- Timeline Track/Clip 的配置方式
- `ComboTag` 驱动逻辑
- `Execute / Buffer / Fallback` 三种窗口语义

### 8.2 升级

升级方向：

1. 从 “Combo 专属” 升级为 “Action Local Command Window”
2. 增强校验
3. 增强可视化调试

### 8.3 需要新增的编辑器优化

建议新增：

- Route 的 `RequiredWindowTag` 下拉枚举，直接读取当前 Timeline 的可用 Tag
- Route 校验：引用了不存在的 Tag 时给出错误
- 窗口预览高亮：当前播放时中枢已激活的 LocalWindow 列表可视化
- 运行时 Debug 面板：显示最近消费的命令、命中的 Route、当前 Context、当前 Action

这样可以在保持现有使用手感的同时，显著提升配置可维护性。

## 9. 运行时职责改造方案

### 9.1 CommandController

建议将现有 `ComboController` 升级为 `CommandController`，并逐步合并职责。

新增职责：

- 支持 `ActionConfigAsset` 级 `LocalRoutes`
- 支持 `CommandContextConfig` 级 `ContextRoutes`
- 支持 `CurrentContext`
- 支持 `CurrentAction`
- 支持统一 `OnActionComplete` 回流策略

保留职责：

- CommandBuffer 生命周期管理
- ComboWindow 生命周期管理
- 窗口标签激活/剥离

### 9.2 State / SubState

重构后：

- 不再直接决定技能入口
- 不再手动调用技能播放
- 只负责：
  - 物理移动
  - locomotion 自动续播
  - 上下文切换

例如：

- `GroundIdle` 进入时只设置 `CurrentContext = GroundIdle`
- `GroundJog` 进入时只设置 `CurrentContext = GroundJog`
- `GroundDash` 进入时只设置 `CurrentContext = GroundDash`

具体要播 `Idle` / `JogStart` / `JogLoop` / `DashStart` / `DashLoop`，依然可由 locomotion 子状态机驱动，但最终播放调用必须走中枢。

### 9.3 ActionPlayer

保持纯播放器定位。

未来要求：

- 除 `HitStun` 等硬中断特殊情况外
- 所有 `PlayAction` 都经由 `CommandController`

## 10. 是否需要独立指令类/接口

## 10.1 不建议做的事情

不建议把运行时命令大规模类化成：

- `BasicAttackCommand`
- `EvadeFrontCommand`
- `IdleCommand`
- `DashAttackCommand`

原因：

- 高频分配无收益
- 序列化与编辑器配置复杂
- 不解决真正的问题
- 会把“枚举膨胀”变成“类型膨胀”

## 10.2 建议接口化的部分

更适合接口化的是：

- `ITransitionCondition`
- `ICommandResolver`
- `IContextSelector`

也就是：

- 运行时命令保持轻量 struct
- 策略与规则接口化

## 11. 分阶段迁移方案

### Phase 1：打通通用 Action LocalRoutes

目标：

- 让 `LocomotionConfigAsset` 也能真正参与 `ComboWindow + Route` 裁决
- 先让 Dash 冲刺普攻跑通

工作项：

- 将控制器内部对当前动作的读取从 `SkillConfigAsset` 改为 `ActionConfigAsset`
- 将 `ActionConfigAsset.OutTransitions` 语义升级为通用 `LocalRoutes`
- 保留现有窗口系统不动

完成标准：

- `DashLoopConfig` 能通过 `ComboWindow + LocalRoutes` 接出冲刺普攻

### Phase 2：统一输入上报

目标：

- 让地面态、冲刺态、后摇态、技能态全部只向中枢上报命令

工作项：

- 收敛 `DefaultInputCommandHandler`
- 收敛 `DashInputCommandHandler`
- 保留统一上报型 handler

完成标准：

- 非受击状态下，所有离散输入都先进中枢，再决定动作

### Phase 3：引入 CommandContext

目标：

- 从“当前动作决定一切”升级为“当前上下文 + 当前动作共同裁决”

工作项：

- 引入 `CommandContextType`
- 在 Ground / Skill / Evade / Backswing / HitStun 切换时写入上下文
- 中枢开始支持 `ContextRoutes`

完成标准：

- Idle / Jog / Dash / Stop 的全局入口不再靠手写 handler

### Phase 4：整合播放控制权

目标：

- 除受击特殊流外，所有 `ActionPlayer.PlayAction` 统一由中枢调用

工作项：

- 中枢接管 Ground locomotion 播放请求
- 子状态只申请上下文/阶段，不直接播

完成标准：

- State / SubState 不再拥有最终播放权

### Phase 5：资产化与图编辑器整合

目标：

- 把 `LocalRoutes + ContextRoutes` 纳入角色级图编辑器

工作项：

- `ActionNode`
- `ContextNode`
- `RouteEdge`
- 编译回 `LocalRoutes / ContextRoutes`

完成标准：

- 一张图管理角色全部动作派生与上下文入口

## 12. 关键风险与规避

### 12.1 风险：重构过大导致战斗手感回退

规避：

- 优先打通 Dash 冲刺普攻这一条路径
- 不先大拆 ComboWindow
- 不先碰全部 Ground 逻辑

### 12.2 风险：中枢变成巨型 God Object

规避：

- `CommandController` 内部分模块：
  - CommandBuffer
  - WindowTracker
  - RouteResolver
  - ContextRuntime
  - ActionExecutor

### 12.3 风险：LocalRoutes 与 ContextRoutes 重复配置

规避：

- 明确规则：
  - 依赖当前动作和时间窗口的，放 LocalRoutes
  - 常驻入口和全局权限，放 ContextRoutes

---

## 13. 实施建议

如果只从工程收益看，推荐开工顺序为：

1. Phase 1：先让 Dash 冲刺普攻在通用 LocalRoutes 跑通
2. Phase 2：统一输入上报
3. Phase 3：引入 Context
4. Phase 4：统一播放控制权
5. Phase 5：做图编辑器整合

最先验证架构正确性的案例，建议就是：

`前闪 -> Dash -> DashAttack`

因为它同时覆盖：

- 局部动作派生
- 上下文续态
- 统一播放控制
- ComboWindow 兼容性

这个案例一旦跑通，后续：

- 普攻连段
- 闪避追击
- 不同 Dash 对应不同 DashAttack
- 特殊技派生

都会自然进入统一体系。
